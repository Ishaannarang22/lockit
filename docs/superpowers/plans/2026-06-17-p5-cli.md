# packages/cli — the `lockit` Binary Implementation Plan (Intended)

> Status: INTENDED — scope-level. Expand into bite-sized failing-test-first TDD steps just-in-time, aligned with the repo state at that time. Plan #1 (docs/superpowers/plans/2026-06-17-p0-scaffold-and-crypto-foundations.md) is the worked example of the target granularity.

**Goal:** Build the `lockit` binary — the universal command surface over `@lockit/core` that lets humans and agents add, organize, link, resolve, and inject secrets, where every agent-facing output is value-free and no secret value ever passes through argv, output, or an agent's context.

**Depends on:** Plan #3 (`@lockit/core` encrypted store + Sets/Slots vault model) and Plan #4 (`@lockit/core` project-world sandbox + human-gated admission + local auth). Transitively depends on Plan #1 / `@lockit/crypto`.

**Packages touched:** `packages/cli` (new). Consumes `@lockit/core` only; never imports `@lockit/crypto` directly (all sealing/unsealing is delegated through `core`, per the inward dependency rule in architecture.md).

---

## Scope — what this subsystem builds

- A single executable `lockit` (bin entry) that parses a verb-first command line and dispatches to `@lockit/core` operations, with a stable, scriptable exit-code contract.
- **Secret commands** over the global store: `secret add` / `set` (create or update a Secret by slug + schema + fields), `rotate`, `ls` (list, value-free), `rename` (records old slug in `aka`), `rm`.
- **Slot/link commands** over the project vault: `link` (declare a slot; `link --set` admits-and-fills in one gesture), `unlink`, and the `slot add` form for declaring a requirement.
- **Resolution + run:** `status` (the onboarding command — lazily resolves every slot and reports OK / missing / open-unfilled / ambiguous), `run` (in-memory injection + masking + shred), and `run --dry-run` (the agent-safe verification primitive: env-var **names** only).
- **Onboarding + identity:** `status` as the headline "what does this project need and what's filled" diff; `whoami` (local identity summary, value-free); `import-env` (ingest an existing `.env` into the store as Secrets, then offer slots — never echoing values back).
- **Sharing stub:** `share` exists as a recognized command that returns a clear "not available until P3" structured error/exit code, so the surface is stable but no crypto is implied here (real sharing is Plan #7).
- **Value entry discipline:** secret values are entered via interactive prompt or piped stdin **only — never via argv** — to keep secrets out of shell history, process listings, and agent transcripts.
- **Agent-safe output formatting:** a shared rendering layer that emits only slugs, schemas, field keys, tags, and `hasValue` booleans, with a `--json` mode whose schema is also value-free, plus masking of any value that could appear in a child process's stdout/stderr at `run`.

- `packages/cli/src/commands/secret-rotate.ts` — `rotate`: new version of a Secret's value(s) via `core`.
- `packages/cli/src/commands/secret-ls.ts` — `ls`: value-free listing with `--schema` / `--tag` filters.
- `packages/cli/src/commands/secret-rename.ts` — `rename`: rename a slug, recording the old in `aka`.

- `packages/cli/src/commands/status.ts` — `status`: lazy resolve all slots; render the OK/missing/open-unfilled/ambiguous onboarding diff.
- `packages/cli/src/commands/run.ts` — `run` and `--dry-run`: drive `core` injection, stream masked child output, shred on exit; dry-run prints names only.

## Key components & responsibilities

**`runCli` + router.** `runCli(argv, io): Promise<ExitCode>` parses global flags, selects a handler, and translates any thrown `CliError` into a stable exit code and a value-free message. Handlers receive a small context `{ io, env, json, core }`. All terminal interaction goes through `io`, never `process.std*` directly, so integration tests can both spawn the real binary and (for unit tests) inject a fake `IO`.

**The `IO` seam and value-input.** Value acquisition is the security-critical surface:

```ts
interface IO {
  readStdin(): Promise<Uint8Array>;
  isStdinTTY: boolean;
  promptHidden(label: string): Promise<string>; // no echo
  confirm(label: string): Promise<boolean>;
  out(s: string): void;
  err(s: string): void;
}

// Refuses argv-borne values by construction: there is no code path that reads a
// secret value from parsed positional/flag args.
async function acquireFieldValue(field: FieldSpec, io: IO, opts: { noInput: boolean }): Promise<string>;
```

`secret add`/`set` parse only field **keys** and metadata from argv; the value side comes from `promptHidden` (TTY) or `readStdin` (pipe). A `FIELD=VALUE` token on argv is a hard error directing the user to prompt/stdin — this is what keeps values out of shell history.

**Value-free rendering.** `render.ts` accepts only the value-free projection that `core` already exposes (slug, schema, aka, field keys + `type` + `hasValue`, tags, version metadata). There is intentionally no rendering function that takes a plaintext value; `--json` serializes the same projection. This makes "agent never sees a value" a structural property of the output layer, not a runtime check.

**Status as the onboarding diff.** `status` asks `core` to resolve every slot for the selected `--env` and renders four states per slot:

```ts
type SlotStatus =
  | { kind: "ok"; schema: string; resolvedTo: string }      // pinned hit, or open auto-filled (chosen slug printed)
  | { kind: "missing"; schema: string; pinnedTo: string }    // pinned slug absent
  | { kind: "open-unfilled"; schema: string }                // open slot, zero candidates
  | { kind: "ambiguous"; schema: string; candidates: ChooserEntry[] }; // N>1, value-free chooser
```

`ChooserEntry` carries `{ index, slug, schema, tags }` only. The exit code is non-zero when any slot is missing/open-unfilled/ambiguous, so a script (or agent) can gate on `status` before `run`.

**`run` and `--dry-run`.** `run` delegates resolution + in-memory decryption + spawn to `core`, pipes the child's stdout/stderr through `mask.ts`, and ensures `core`'s shred-on-exit runs. `--dry-run` never decrypts or spawns: it prints the env-var **names** that would be set (values masked/omitted) and flags duplicate inject names, unfilled open slots, and ambiguity — the agent-safe preview.

**Ambiguous resolution as a structured error.** When `core` returns `AMBIGUOUS`, the CLI surfaces it identically in human and `--json` modes: a numbered, value-free chooser plus a dedicated exit code, so an agent must surface it to a human rather than guess.

## Tests that prove it — security properties emphasized

These are **integration tests that spawn the real compiled `lockit` binary** via `spawn-lockit.ts` (temp HOME, temp project), plus focused unit tests on the `IO`/render seams. Each is one intention:

- **Values never come from argv (no-shell-history property):** invoking `secret add openai/dev OPENAI_API_KEY=sk-leak` exits non-zero with an error pointing to prompt/stdin, and nothing is stored — proving the only value path is prompt/stdin.
- **stdin value path works and leaks nothing:** piping a value into `secret add` stores it (verified later by `run` injecting it into a child), while the command's own stdout/stderr contain no occurrence of the value.
- **`ls` is value-free:** after storing multi-field Secrets, `ls` (and `ls --json`) output contains slugs/schemas/field keys/`hasValue`/tags and never any stored value, masked or otherwise.
- **`status` onboarding diff is correct and value-free:** against a vault with one pinned-hit, one pinned-missing, one open-unfilled, and one open-auto-filled slot, `status` reports exactly OK/missing/open-unfilled and prints the auto-chosen slug for the single-candidate open slot, with a non-zero exit when anything is unresolved — and no values appear.
- **Ambiguous resolution is a hard structured error:** with two Secrets of the same schema and an open slot, `status`/`run --dry-run` emit the `AMBIGUOUS` structured error with a numbered chooser listing slug/schema/tags only (no values), and a dedicated exit code — the model cannot resolve it by guessing.
- **`run --dry-run` is the agent-safe preview:** dry-run prints the exact env-var **names** (including one-value-many-names expansion) without decrypting or spawning, and surfaces a duplicate-inject-name collision as a hard error — output contains zero values.
- **Injection isolation at `run`:** `lockit run -- <printenv-style child>` shows the child receives the resolved env vars, but the env vars are **not** present in the parent/`lockit` process environment, and `lockit`'s own (non-child) output never contains the value.
- **Output masking at `run`:** a child that deliberately prints its secret to stdout/stderr has that value masked in what `lockit run` emits, proving the masking chokepoint is active on the child stream.
- **File-type field path safety:** a `type:"file"` field is materialized only for the child (path in the env var), the contents never appear in `lockit`'s output, and the temp path is gone after exit.
- **`link` enforces the unique-inject-name invariant:** linking a second slot that injects an already-claimed env-var name fails at link time with a clear value-free error.
- **`rename` keeps references resolving:** renaming a slug records the old slug in `aka`, and a slot pinned to the old slug still resolves.
- **`import-env` never echoes values:** importing a `.env` creates the expected Secrets but the command output (and any `--json`) shows only keys/slugs/`hasValue`, never the imported values.
- **`share` is a stable deferred stub:** `lockit share ...` exits with the documented "not available until P3" structured error and does not touch the store.
- **Exit-code contract:** a table-driven test pins each outcome (success, missing, open-unfilled, ambiguous, deferred, usage error) to its stable exit code so scripts and agents can branch on them.

## Open questions

- **`--json` schema versioning:** should the value-free JSON output carry a `schemaVersion` from day one so agents can pin to it?
- **Hidden-prompt portability in CI:** confirm `promptHidden` degrades correctly to stdin when there is no TTY, and that integration tests exercise both branches across macOS and Linux runners.
- **`import-env` slug inference:** how aggressively to infer slugs/schemas from `.env` var names without crossing into "guessing" (must stay consistent with the never-guess resolver philosophy).
- **`link --set` granularity:** whether the combined declare-then-admit gesture should batch multiple slots into one admission confirmation, mirroring `core`'s batch-admission box.
- **Masking robustness:** how to mask secret values in child output that the child transforms (e.g. base64/url-encodes) — documented as a best-effort limit consistent with the honest "containment is not omnipotence" stance, with a test pinning the plain-substring guarantee only.
