# Spec — 0.4.0: per-project keys + cross-platform admission + agent plugin

Status: approved design (chat), ready to build
Date: 2026-06-25
Supersedes: the global-only resolution behavior; adjusts the roadmap's Sets+Slots
toward a simpler per-project binding model.

## Why

Today there is only a **global store** keyed by slug, and `run`/`pull` resolve a
bare variable name across everything. Two projects that each want `DATABASE_URL`
with different values collide — there is no notion of "in *this* project,
`DATABASE_URL` means *this* value." This release makes a project track its own
keys, gates bringing a stored secret into a project behind human auth, and ships
a Claude Code plugin so agents can drive it safely.

## Data model

Two layers:

- **Global store** — `~/.lockit/store.json` (encrypted). Holds the **values**, the
  single source of truth. Secrets are bags of fields keyed by slug (unchanged).
- **Project vault** — `./.lockit/vault.json` (value-free, committable). For one
  project, a map of **env-var name → reference to a stored secret field**:

  ```jsonc
  // projectA/.lockit/vault.json
  { "DATABASE_URL":   "projectA/db#DATABASE_URL",
    "OPENAI_API_KEY": "openai/personal#OPENAI_API_KEY" }  // shared, reused

  // projectB/.lockit/vault.json
  { "DATABASE_URL":   "projectB/db#DATABASE_URL" }         // same name, different value
  ```

"Same name, different value per project" falls out: each project binds the name
to its own stored secret. Reuse is preserved: many vaults can bind to one shared
secret (rotate once, every project follows). The **name is per-project** (vault);
the **value lives once** (store). The vault is value-free, so it is safe to commit.

**Project identity.** A project is the nearest ancestor directory containing
`./.lockit/`, created by `lockit init` (like `git init`). No daemon, no watcher;
resolution is lazy, only at `run` / `pull` / `status`.

**Project-local vs shared secrets.** `lockit set NAME` inside a project creates a
**project-local** secret (slug namespaced to the project, e.g. `@proj/<dir>/NAME`)
and binds it in the vault. Reuse of an existing/shared secret is the explicit
`lockit admit <slug> --as <NAME>`. Pre-existing global secrets keep working as
shared secrets you can `admit`.

## Admission + sandbox

- **The vault's bindings are the admitted set.** `run`/`pull` may only use keys
  bound in the current project's vault. An unbound name is a hard, value-free
  error — never a global guess.
- **Admission = adding a binding to an existing stored secret**, and it is the
  only gated action: a human presence prompt on `/dev/tty`
  — `Allow "OPENAI_API_KEY" → openai/personal for this project? [y/N]` — which an
  agent driving stdin cannot answer. Approved once, recorded in the vault,
  agent-first thereafter (the lighter policy).
- **`lockit set NAME` (new project-local value, supplied via stdin) is NOT gated**
  — supplying a fresh value is not gaining access to a stored one.
- **Strict 0/1/N resolver** runs at *admission* time: `admit --schema openai`
  with exactly one match auto-resolves (and prints which); 0 or N is a hard
  `AMBIGUOUS`/`not found` error with a value-free chooser. `run` uses the explicit
  vault bindings, so it is never ambiguous.
- **`AuthProvider` seam.** The presence gate is an interface; the `/dev/tty`
  provider is v1, Touch ID / OS-password is the 0.5.0 provider — no rework.

## Agent-safe surfaces

- `lockit status` — the current project's bound keys and their admitted state,
  value-free (name, bound slug/field, `hasValue`). Never a value.
- `lockit run --dry-run` — the env-var names that would be set, value-free; flags
  unbound names and duplicates.
- All agent-facing output stays value-free; values flow only via `run` injection
  (in memory, masked) or the explicit, human/`--yes` `pull` to disk.

## The plugin (P2)

`plugin/` — a Claude Code plugin:
- **Skill** — drive lockit by names only: `status` / `ls` / `--dry-run` to see
  structure, `run` to *use* a secret without seeing it, *request* admission and
  let the human approve, never request or print a value.
- **Hooks** — an egress guardrail that warns when a raw secret looks about to be
  written into a file or command.

## Scope / non-goals (this release)

- **In:** per-project vault + project-local/shared secrets, admission + sandbox,
  strict resolver + chooser, `init` / `admit` / `status` / `--dry-run`, vault-aware
  `run`/`pull`, the plugin. Cross-platform; `crypto` unchanged.
- **Out (→ 0.5.0):** native macOS Touch ID / Secure Enclave keychain cache and the
  per-use/per-session policy; the DEK-indirection crypto envelope change rides
  with it. The 0.4.0 presence gate is `/dev/tty`.
- **Out (later phases):** identity/device enrollment, end-to-end sharing, the team
  server. Account recovery remains a non-goal.

## Honest limits (documented, not hidden)

- The store key still lives on disk (auto keyfile) in 0.4.0, so a local process
  could read it directly and bypass the `/dev/tty` gate. The gate is real against
  an agent that can't answer `/dev/tty`, but the cryptographic teeth (key behind
  Touch ID) land in 0.5.0. Stated plainly in README + help.
- Containment is not omnipotence: once a key is admitted and injected, the child
  process holds the value and could leak it through a command it was allowed to run.

## Success criteria

1. Two projects each bind `DATABASE_URL` to different values; `run` in each injects
   the right one; neither can see the other's.
2. `run`/`pull` refuse an unbound name (value-free error), and refuse a bound name
   whose secret was never admitted.
3. `admit <existing secret>` prompts on `/dev/tty`; an agent (no tty) cannot
   self-admit; approval records the binding and is not re-prompted.
4. `lockit set NAME` (new project-local) works with no prompt.
5. No agent-facing surface (`status`, `ls`, `--dry-run`, chooser, errors) ever
   prints a value.
6. The plugin skill + egress hook load in Claude Code and drive the flow.
