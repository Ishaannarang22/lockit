A local vault your agents
can use but never see.

lockit stores every API key once, encrypted on your machine, and injects it straight into the commands that need it — so a secret never touches your shell history, a committed .env, or an AI agent's transcript.

$ npm i -g @lockit/cli

No account. No server. The first command creates your encrypted store at ~/.lockit.

The differentiator

Agents request. Agents use.
Agents never see.

An AI coding agent needs your keys to build and test — but the moment a value lands in its context, it's in a transcript you can't un-write. lockit gives the agent a way to use a secret without the value ever crossing into what it can read.

Without lockit value leaks into the transcript

01

Value-free output

Every agent-facing command shows slugs, field names and hasValue — the structure of your secrets, never the secrets.

02

Request, don't take

An agent can ask to admit a key with lockit admit, but only a present human can approve it. It can never pull from the store itself.

03

Ships with a skill

lockit install writes an agent-safe skill to ~/.claude/skills, so Claude drives lockit by name and knows the rules in every repo.

The vault

One store. Every project. Zero copies.

Encrypted at rest

Client-side XChaCha20-Poly1305 with Argon2id key derivation, via libsodium. Store and key files are 0600. No operator, no master key.

Inject, don't paste

lockit run -- <cmd> decrypts in memory, sets the child's env, and masks values in its output. Nothing hits your history or a committed file.

Per-project admission

A project can only use keys you've admitted to it — gated by a macOS Touch ID dialog an agent can't satisfy. The binding map is value-free and committable.

Same name, per project

DATABASE_URL can hold a different value in every project. References, not copies — rotate once and every project follows.

End-to-end sharing

Send a secret to a teammate with X25519 / HPKE. An optional relay carries only ciphertext — no operator ever holds a key that opens it.

Zero setup

No passphrase to export, no config to write. The store keys itself with a machine-local key kept in the keychain behind Touch ID.

A day with lockit

From scattered .env files to one clean run.

  1. Store it once

    The value comes from stdin only — never argv — so it stays out of ps and your shell history.

  2. See it, value-free

    lockit ls --vars lists what exists: names, schemas, hasValue. Safe to run in front of anyone — or any agent.

  3. Admit into a project

    lockit init, then admit exactly the keys this repo should use. One Touch ID prompt, then it's bound.

  4. Run with it injected

    lockit run -- npm start puts the real values in the child's environment, masks them in output, and shreds on exit.

Teams

          
Share a secret without trusting a server with it.

          

            Sharing is sealed to your teammate's public key with X25519 / HPKE before it ever
            leaves your machine. The relay — the public one, or one you host — only ever
            carries ciphertext. There is no operator master key, because there's nothing to hold.
          

          
    
            
  • Client-side encryption; the relay is a dumb pipe for sealed bytes.
    • 
            
  • Confirm a teammate's identity out-of-band with lockit identity whois.
    • 
            
  • Bring your own relay: lockit relay set https://relay.mycorp.example.
    • 
          

        

      

    


    
    

      

        
Honest by design

        
Limits documented, never hidden.

        

          lockit is early (0.x) and says so. A child process holds the real value while
          it uses it, so containment isn't omnipotence. Node can't guarantee wiped memory. And with
          zero-knowledge encryption, a passphrase you set and lose is unrecoverable. The threat model
          spells all of this out — read it before you trust it.
